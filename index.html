01 · Research
Before We Started

Our Research Foundation

Before writing code, we spent time understanding the core problems: multi-brand healthcare apps, dynamic UI, and cross-platform performance.

🎯 Problems We Identified

Multiple Brand Identities

Each clinical program needs its own app store listing, branding, colors, and domain-specific features.

Slow UI Iteration Cycles

Native-only apps require full App Store/Play Store reviews just to change a button color or text layout.

Code Duplication Risk

Without a shared foundation, the same bug would need to be fixed in 3 separate repos. Technical debt compounds.

Feature Parity Drift

Separate codebases diverge within months — one app gets a feature, others don't. Users get inconsistent experiences.

✅ Key Research Conclusions

Single Codebase > Multiple Repos

One repo with flavor/scheme separation eliminates drift. Bug fix once → all apps updated instantly.

SDUI for Dynamic Layouts

Backend-controlled JSON templates let backend/design teams push UI changes without mobile deploys.

Flutter & Stac SDUI Engine

100% native Flutter widgets compiled dynamically at runtime from JSON configurations using the Stac engine.

Riverpod for State Management

Compile-time safe, testable, no boilerplate. Perfect for a clean architecture with feature isolation.

🧱 Three Architectural Pillars

1. Unidirectional Data Flow
UI watches Riverpod providers → providers pull from repos → repos call APIs → events go back up. Never the other direction.
2. Stateless UI Layer
Widgets never fetch network data or handle business logic. They are purely reactive — they watch providers and rebuild when data changes.
3. Decoupled Business Logic
Providers handle operations (optimistic states, navigation triggers) and map data models into clean SDUI layout maps.

🏗️ Architecture Layers & System Dependencies

This blueprint illustrates DoctorPoint's strict vertical layering. The Presentation Layer (containing standard Views, Widgets, and Stac-based SDUI templates) remains entirely decoupled from state management, delegating to Riverpod State Providers (such as HydratedStateNotifier). State providers query Repositories and Data Sources in the Data Layer, which route calls through Infrastructure utilities—specifically SharedPreferences, dynamic Stac Parsers, the HTTP File Cache, and our custom Dio-based ApiService—before fetching from Backend Endpoints.

🔐 Router Guards & Global Redirect Lifecycle

This flow details MaterialApp.router route guarding. Upon initiation, goRouterProvider runs a Global Redirect Check: if a user attempts to access a premium path without an appropriate subscription tier, they are dynamically redirected to /premium-features; if no active user session exists, they are routed to /login. Similarly, when the network interceptor detects a session expiration, the onSessionExpired callback forces a redirect to the Login screen. Authorized sessions bypass guards, mounting inside the ShellRoute BottomNavigationWrapper to resolve core paths: `/dashboard`, `/consultation`, `/progress`, `/chat`, `/profile`, and `/community`.

⚡ Network Interceptor Sequence & Token Refresh Pipeline

This sequence documents asynchronous network mutations. The View watches/mutates data through the Repository, calling the Dio ApiService. The request runs through continuous interceptors: (1) Connection Check verifies connectivity; (2) Dio Cache TTL intercepts and returns cached responses when applicable; (3) TokenStorage Bearer injection resolves and appends the auth token; (4) Response Code Validation asserts 200/201/304 status before dispatching the payload. If a 401 Unauthorized error occurs, a refresh is triggered via the `AuthRepository` callback: if the refresh completes successfully, the ApiService retries the request once; if the refresh fails, local session tokens are cleared, and `onSessionExpired` redirects the user back to `/login`.

04 · Server-Driven UI
The SDUI Engine

What is Server-Driven UI & How It Works

SDUI lets the backend control what the app looks like. Instead of hardcoding layouts in the app, we send JSON from the server that describes the UI — the app just renders it.

🚀 Live Interactive SDUI Engine Sandbox

Want to see this in action? Play with our real-time Stac JSON compiler and watch widgets render instantly on a simulated mobile device.

❌ Traditional Approach

Change button text → write code → PR review → App Store submit → wait 2–5 days → users update app
Change card layout → same process × 3 apps
A/B test a new screen → deploy 2 app versions

✅ With Server-Driven UI

Change button text → update JSON on backend → all users see it instantly
Change layout → update JSON → live immediately
A/B test → serve different JSON to user segments

⚙️ The 4-Stage Processing Pipeline

Every layout JSON goes through 4 stages before rendering. This is what transforms a raw template into a beautiful, localised, responsive, branded screen.

STAGE 01
📦

DataLoader

Load raw JSON from asset or API. Inject dynamic vars like {{userName}}

STAGE 02
🌍

Localizer

Find all $tr: keys and replace with active language translations

STAGE 03
📐

Responsive

Convert .rw/.rh/.rsp suffixes into device-specific logical pixels

STAGE 04
🎨

Themer

Resolve $color: keys using program brand colors (Weight Loss vs Diabetes etc.)

FINAL
🚀

Stac Render

Pass processed JSON to Stac engine → native Flutter widgets rendered

Multi-Program & Brand Strategy

White-Label & Flavor Architecture Strategy

Transitioning from a single-app model to a highly scalable, multi-brand platform (HealthyBMI + Diabetic Free Forever - DFF) leveraging a 70% shared core and a 30% brand-specific layer.

RECOMMENDED

✅ Single Codebase + Multi-Flavor + SDUI

✓ 70% code reuse (Auth, Profile, Community, Chat, Consultation)
✓ Zero SDK drift: All brands compile on the same framework versions
✓ Bug-fix once: Core logic updates automatically deploy to all apps
✓ Dynamic customization: BrandConfig + SDUI gates features at runtime
✓ DevOps efficiency: Single codebase, unified pipeline, simplified certs
NOT RECOMMENDED

❌ Multiple Standalone Codebases

✗ Massive copy-paste overhead to maintain identical modules
✗ Inevitable feature drift: App features diverge within 3–6 months
✗ Tripled maintenance: Same bug must be patched separately in 3 repos
✗ High DevOps load: 3 distinct pipelines, 3 sets of store credentials
✗ Slow launches: Spinning up a new program takes weeks or months

📋 HealthyBMI & DFF Module Split Strategy

To maximize velocity, we partition the app into a Shared Module Layer (~70%) and a Brand-Specific Customization Layer (~30%). Features in the shared block remain completely unified.

📦 Shared Core Modules (~70%)
  • Authentication: OTP dispatch, token acquisition, refresh loops
  • Profile Management: Demographics, onboarding setups
  • Community & Forums: Clinical social feed, posts, comments, likes
  • In-App Chat: Real-time clinical expert messaging & WebSockets
  • Consultation: Video appointments, custom Meeting WebView, calendars
  • Health Assessment: Standard medical questionnaire engine
  • Knowledge Base & Courses: Clinical educational material player
  • Infrastructure: ApiService, SharedPreferences, Push services
🎨 Brand-Specific Modules (~30%)
  • Home Dashboards: Brand-specific SDUI template JSON maps
  • Trackers:
    • HealthyBMI: Diet, medication, exercise, weight trackers
    • DFF: Glucose logging, specialized meal plans, active sessions
  • Monetization & Plans: Subscriptions, enrollment, checkouts
  • Marketing Elements: Announcements, campaigns, webinars
  • Brand Assets & Copy: Splash screens, icons, localized brand terms
  • AI Tools: Health chatbot integration specific to program guidelines

⚙️ Compile-Time Defines & Brand Config

Build tools pass brand tokens at compile time via --dart-define parameters. A centralized BrandConfig injects specific themes, APIs, and feature sets at startup. 

// lib/app/brands/brand_config.dart
abstract class BrandConfig {
  AppBrand get brand;
  String get appName;
  Set<FeatureId> get enabledFeatures;
  List<NavigationItem> get navItemsFreeUser;
  List<NavigationItem> get navItemsPaidUser;
  String get defaultHomeRoute;
  BrandTheme get theme;
  BrandApiConfig get api;
  String assetPath(String file) => "assets/brands/${brand.name}/$file";
}

An AppBrand provider overrides Riverpod scopes. The application is completely brand-agnostic outside this boundary!

Multi-Tenant API Tenancy: Either passes a custom header X-App-Brand: dff on a single host client, or routes to distinct brand subdomains based on BrandConfig.api configurations.

⚡ Dynamic Runtime Feature Gating & Conditional Flow

White-label logic does not require duplicative code or messy compile-time preprocessors. By referencing the compile-time injected BrandConfig or active flavor properties, we cleanly switch UI elements and feature modules at runtime. 

// Example: Gating brand-specific reports in shared screen wrapper
Widget buildReportSection(BuildContext context, WidgetRef ref) {
  final brand = ref.watch(brandConfigProvider).brand;

  if (brand == AppBrand.healthybmi) {
    // HealthyBMI flavor: show high-priority lipid details
    return CholesterolReportCard(
      title: "Lipid Profile",
      showHdlLdl: true,
    );
  } else if (brand == AppBrand.dff) {
    // Diabetic Free Forever (DFF): show blood glucose metrics
    return SugarReportCard(
      title: "HbA1c & Blood Sugar Tracker",
      showLogs: true,
    );
  }

  return SizedBox.shrink();
}
💡 How Gated Branching Works

Instead of spreading conditional blocks throughout our entire layout structure, we isolate them inside small, modular presentation widgets:

  • Riverpod Provider State: brandConfigProvider is loaded during bootstrap and read at compile time/startup to provide flavor context.
  • Strict Encapsulation: The main page shell (like the Reports Screen) is shared between both apps, keeping the layout code perfectly unified.
  • Conditional Insertion: The minor brand differences (e.g. CholesterolReportCard vs SugarReportCard) are dynamically loaded side-by-side inside safe widget slots.
🚀 Ensures 70%+ codebase sharing while giving total design and feature autonomy!

🔐 Feature Registry & Router Guards

Rather than using messy compilation directives (#ifdef), a centralized registry evaluates brand privileges dynamically. Unsupported pages are completely omitted from the router map or redirected immediately. 

// lib/app/brands/feature_registry.dart
enum FeatureId {
  profile, community, chat, consultation,
  dietTracker, glucoseTracker, mealPlan,
  subscription, aiTools
}

class FeatureModuleRegistry {
  final BrandConfig config;
  FeatureModuleRegistry(this.config);

  bool isEnabled(FeatureId id) =>
      config.enabledFeatures.contains(id);
}
Storage Namespacing: Keys are prefixed dynamically with {brand}_ (e.g. dff_auth_access_token) to prevent data leakages if multiple programs reside on the same testing hardware.
